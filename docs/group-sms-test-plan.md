# Group SMS 문자 발송 테스트 플랜

## 목적

이 문서는 `소희야 가자` 단체문자 기능을 실제 고객에게 적용하기 전에 단계별로 검증하기 위한 실행 계획이다.

목표는 다음과 같다.

- 단톡방 없이 수신자별 1:1 메시지가 생성되는지 확인
- 전화번호와 개인화 본문이 다른 고객과 섞이지 않는지 확인
- 반복, 딜레이, 취소, 재실행 과정에서 중복 발송이 발생하지 않는지 확인
- iOS Shortcuts와 앱 callback 상태를 구분해 기록
- 사진/파일 첨부 시 iMessage, RCS, MMS 차이를 확인
- Android 포팅 후 같은 테스트 데이터를 재사용

실제 전화번호, 고객명, 수신 메시지 스크린샷은 Git에 커밋하지 않는다. 익명화된 결과와 통계만 저장소 문서에 기록한다.

## 테스트 원칙

1. Gate A~C는 사용자 본인 소유 번호만 사용한다.
2. 실제 고객 번호는 Gate D 승인 전 사용하지 않는다.
3. 한 단계가 실패하면 다음 단계로 확대하지 않는다.
4. 예상하지 않은 번호가 한 번이라도 나타나면 즉시 단축어를 중단한다.
5. 통신사 경고 문자가 오면 해당 날짜의 시험을 중단한다.
6. 앱의 `발송 요청 완료`와 실제 수신자의 `수신 완료`를 별도로 기록한다.
7. 수신 결과를 앱이 자동으로 안다고 가정하지 않는다.
8. 200건·500건 제한을 확인하려고 실제 한도까지 보내는 스트레스 테스트는 하지 않는다. 정책 보호선은 단위 테스트와 모의 데이터로 검증한다.

## 공통 시험 데이터

실제 번호는 앱에만 입력하고 문서에는 아래 별칭을 사용한다.

| 별칭 | 용도 |
| --- | --- |
| `TEST-A` | 사용자 보조 번호 1 |
| `TEST-B` | 사용자 보조 번호 2 |
| `TEST-C` | 필요 시 사용자 소유 추가 번호 |

기본 본문:

```text
소희야 가자 단체문자 시험 {순번}/{전체} - {반복}
```

문자열 검증용 본문:

```text
[TEST-{순번}] 한글 ABC 123 !@# 줄바꿈
두 번째 줄
```

테스트 번호는 서로 명확하게 구분되는 번호를 사용하되 결과 문서에는 마지막 자리도 남기지 않는다.

## 결과 상태

각 시험은 다음 중 하나로 기록한다.

- `PASS`: 모든 기대 결과 충족
- `FAIL`: 기대 결과와 다름
- `BLOCKED`: 권한, 통신사, 기기 상태 때문에 실행 불가
- `PARTIAL`: 일부만 확인 가능
- `NOT RUN`: 아직 실행하지 않음

## Gate A - 데이터 전달 검증

실제 문자를 보내지 않고 앱과 단축어의 데이터 연결만 확인한다.

### A-01 payload 생성

입력:

- 번호: `TEST-A`
- 번호당 반복: 1회
- 딜레이: 꺼짐

절차:

1. 앱에서 테스트 모드를 선택한다.
2. 번호와 본문을 입력한다.
3. `payload만 클립보드에 복사`를 누른다.
4. 단축어 검증본 `SoheeGroupSMS-ReadTest`를 실행한다.

기대 결과:

- `PhoneNumber`가 TEST-A의 실제 번호와 일치
- `MessageBody`가 앱 미리보기와 일치
- 문자는 발송되지 않음

현재 상태: 2026-07-12 `PASS`

### A-02 배열 순서

입력:

- 번호: `TEST-A`, `TEST-B`
- 번호당 반복: 2회

기대 순서:

```text
TEST-A 반복 1
TEST-A 반복 2
TEST-B 반복 1
TEST-B 반복 2
```

앱 미리보기, payload, 단축어 표시 순서가 모두 같아야 한다.

현재 상태: 2026-07-12 반복 확대 흐름 `PASS`, 정확한 순서 증거는 후속 기록 권장

## Gate B - 자동 텍스트 발송 검증

Gate B는 사용자 본인 번호로만 실행한다.

### B-01 단일 번호 1건

입력:

- 번호: `TEST-A`
- 반복: 1회
- 딜레이: 꺼짐

- TEST-A에서 본문 1건 수신
- 다른 번호에는 수신되지 않음

현재 상태: 2026-07-12 `PASS`

### B-02 여러 번호 1회

입력:

- 반복: 각 1회

기대 결과:

- TEST-A와 TEST-B에 각각 1건
- 그룹 대화가 생성되지 않음
- 각 대화의 수신자가 1명

현재 상태: 2026-07-12 `PASS`

### B-03 여러 번호 반복

입력:

기대 결과:

- TEST-A 3건, TEST-B 3건
- `{순번}`, `{전체}`, `{반복}` 값이 기대 순서와 일치
- 누락, 추가, 중복 없음

현재 상태: 2026-07-12 `PASS`

### B-04 한글과 장문

입력:

- 한글, 영문, 숫자, 기호, 줄바꿈 포함
- SMS 길이를 넘는 장문 1건

기대 결과:

- 인코딩 손상 없음
- 줄바꿈 유지 여부 기록
- 메시지 앱에서 SMS/LMS/MMS/RCS/iMessage 중 어떤 형태로 처리됐는지 기록
- 통신사 과금 단위를 앱이 단정하지 않음

현재 상태: `NOT RUN`

## Gate C - 딜레이와 제어 흐름

### C-01 고정 딜레이

입력:

- 번호: `TEST-A`, `TEST-B`
- 반복: 각 2회
- 고정 딜레이: 5초

준비:

- 단축어에서 `plannedDelaySeconds`를 `DelaySeconds`로 추출
- `DelaySeconds > 0`이면 `대기` 실행

측정:

- 각 메시지의 발송 또는 수신 시각
- payload의 `plannedDelaySeconds`

허용 기준:

- Shortcuts와 통신망 처리 오차를 감안해 관찰 간격이 계획값보다 짧아지지 않는지 확인
- 수신 시각만으로 정확한 단말 발송 시각을 단정하지 않음

현재 상태: `NOT RUN`

### C-02 랜덤 딜레이

입력:

- 최소 2초, 최대 6초
- 총 6건 이하

기대 결과:

- 모든 payload 값이 2~6초 범위
- 첫 항목의 정책상 딜레이가 0이라면 명세와 일치
- 실제 실행이 저장된 계획값 순서를 따름

현재 상태: `NOT RUN`

### C-03 딜레이 꺼짐

기대 결과:

- 모든 `plannedDelaySeconds`가 0
- 단축어 자체 처리 시간은 발생할 수 있음

현재 상태: 앱 payload 생성과 빠른 반복 발송 `PASS`

### C-04 완료 callback

절차:

1. 1~2건 발송을 완료한다.
2. 단축어 마지막에서 완료 callback URL을 연다.
3. 앱의 최근 캠페인을 확인한다.

기대 결과:

- 정확한 `campaignId`의 상태만 `발송 요청 완료`로 변경
- 고객 히스토리에 실제 전달 성공이 아니라 `단체문자 요청`으로 기록

현재 상태: `NOT RUN`

### C-05 사용자 취소

절차:

1. 딜레이가 있는 소규모 시험을 시작한다.
2. 중간에 Shortcuts 실행을 사용자가 취소한다.

기대 결과:

- 미발송 항목이 자동으로 다시 시작되지 않음
- 앱 상태가 `사용자 취소` 또는 `확인 필요`
- 이미 요청한 항목을 자동 재전송하지 않음

현재 상태: `NOT RUN`

### C-06 앱/단축어 중단과 재실행

검증 항목:

- 화면 잠금
- 앱 전환
- 단축어 강제 종료
- 네트워크 단절
- 기기 재부팅 후 캠페인 조회

중단된 캠페인은 전체 재실행 버튼을 즉시 제공하지 않고 사용자가 수신 결과를 확인하게 해야 한다.

현재 상태: `NOT RUN`

## Gate D - 첨부파일

Gate D는 사진 1장부터 시작한다.

### D-01 iPhone 사진 1장

입력:

- TEST-A 1명
- JPEG 또는 HEIC 사진 1장
- 짧은 본문

기대 결과:

- 정확한 수신자 한 명에게만 전송
- 본문과 사진이 한 메시지인지 두 메시지인지 기록
- 사진 방향과 품질 확인
- iMessage/MMS/RCS 유형 기록

현재 상태: `NOT RUN`

### D-02 사진 여러 장

입력:

- TEST-A 1명
- 서로 다른 JPEG/HEIC/PNG 사진 2~5장
- 짧은 본문

기대 결과:

- 선택한 사진이 모두 같은 수신자에게 전달됨
- 앱에서 정한 첨부 순서가 유지됨
- 일부 사진만 누락되거나 이전 사진이 중복되지 않음
- 전체 크기 초과 시 사전점검 또는 명확한 오류 표시

현재 상태: `NOT RUN`

### D-03 여러 번호에 같은 다중 사진

입력:

- TEST-A, TEST-B
- 사진 2장 이상
- 각 1회

기대 결과:

- 각 번호에 동일한 사진 개수와 순서
- 그룹 대화 없음
- 이전 반복의 첨부가 다음 수신자에서 누락되지 않음

현재 상태: `NOT RUN`

### D-04 Android 수신자의 RCS/MMS

검증 항목:

- RCS 켬/끔
- 고해상도 사진의 압축 결과
- 데이터 네트워크가 꺼진 경우
- 실패 시 단축어와 앱이 알 수 있는 정보 범위

현재 상태: `NOT RUN`

### D-05 일반 파일 여러 개

파일 후보:

- 작은 PDF 여러 개
- 문서 파일과 이미지 파일의 혼합
- 지원 여부 확인용 텍스트 파일

기대 결과:

- 지원하지 않는 전송 방식에서 조용히 성공으로 기록하지 않음
- 파일 크기와 MIME type 제한이 UI에 표시됨
- 모든 파일의 개수와 순서가 수신 결과와 일치함

현재 상태: `NOT RUN`

### D-06 사진과 파일 혼합

입력:

- TEST-A, TEST-B
- 사진 2장
- PDF 또는 텍스트 파일 1개 이상
- 본문 있음/없음 각각 시험

기대 결과:

- 각 수신자에게 같은 혼합 첨부 목록 전달
- 특정 전송 방식에서 혼합 첨부가 불가능하면 발송 전 차단 또는 명시적 분할 안내
- 취소 후 임시 첨부가 다음 캠페인에 섞이지 않음

현재 상태: `NOT RUN`

## Gate E - 정책과 안정성

### E-01 앱 보호선 단위 테스트

실제 문자를 보내지 않고 다음 경계를 모의 데이터로 검사한다.

```text
149, 150, 179, 180, 199, 200, 400, 499, 500
```

각 값에서 경고, 명시적 확인, 차단 결과가 문서화된 정책과 일치해야 한다.

### E-02 소규모 장시간 시험

실제 한도 시험이 아니라 사용자 본인 번호에 제한된 소규모 시험으로 다음을 본다.

- 총 실행 시간
- 기기 발열
- 배터리 소모
- 화면 잠금 영향
- Shortcuts 중단 여부
- 누락/중복

초기 상한은 10건 이하로 두고 결과를 검토한 뒤 확대한다.

### E-03 통신사 경고

통신사 경고 문자, 과금 안내, 발송 제한이 발생하면:

1. 즉시 시험 중단
2. 해당 날짜 추가 시험 금지
3. 로컬 증거 보관
4. 앱 카운터와 실제 발송 추정치 비교
5. 정책 설정 재검토

## Gate F - 제한된 고객 파일럿

진입 조건:

- Gate A~C 모두 PASS
- 첨부를 사용한다면 해당 Gate D 항목 PASS
- callback과 취소 흐름 PASS
- 개인정보와 수신 동의 기준 확인
- 메시지 템플릿 최종 검토

파일럿 원칙:

- 명확하게 선정된 소수 고객
- 한 캠페인 한 번만 전송
- 발송 직전 대상과 본문 미리보기 저장
- 고객별 제외 가능
- 즉시 중단 가능
- 결과를 `전달 성공`이 아니라 `발송 요청`으로 기록

## 즉시 중단 조건

아래 중 하나라도 발생하면 현재 캠페인을 중단한다.

- 예상하지 않은 전화번호가 표시되거나 수신
- 서로 다른 고객의 이름, 번호, 본문이 섞임
- 그룹 대화 생성
- 계획보다 많은 메시지 발송
- 단축어를 다시 열었을 때 이전 payload 재전송
- 사용자 취소 후에도 계속 발송

## Android 재사용 기준

Android에서도 Gate A~F의 입력과 기대 결과를 그대로 사용한다. 달라지는 것은 전송 어댑터와 관찰 가능한 결과다.

- `SmsIntentTransport`: 사용자의 전송 탭이 필수이므로 자동 반복 항목은 지원 불가로 기록
- `DefaultSmsTransport`: sent/delivery `PendingIntent`를 각각 기록
- 다중 SIM: 선택된 `subscriptionId` 확인
- MMS: APN, 데이터 연결, 기본 SMS 역할과 파일 크기 기록
- 앱 재시작: Room에 저장된 idempotency key로 중복 발송 방지 확인

Android에서 실제 자동 SMS 시험은 Google Play 권한 정책과 배포 방식을 확정한 뒤 진행한다.
