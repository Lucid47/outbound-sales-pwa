# 네이티브 앱 릴리스 대장

이 문서는 `소희야 가자` iOS/iPadOS 앱의 TestFlight 빌드와 Git 소스를 연결합니다. 날짜를 기억해 원복하지 않고 `앱 버전`, `빌드 번호`, `Git 태그`를 기준으로 판단합니다.

## 번호 읽는 법

- 앱 버전: 사용자에게 보이는 제품 버전입니다. 예: `1.0`
- 빌드 번호: 같은 앱 버전 안에서 업로드할 때마다 증가하는 내부 번호입니다. 예: `3`, `4`
- 표시 방식: `1.0 (4)`는 앱 버전 1.0의 네 번째 빌드를 뜻합니다.
- Git 태그: 실제로 업로드한 소스를 다시 찾는 고정 이름입니다. 예: `testflight/1.0-build-4`

## 1.0 (3)

기준 소스:

```text
Git 태그: testflight/1.0-build-3
Git 커밋: a22670170b579d05772a75aa5f1bca544a004ea2
```

포함 기능:

- iPhone/iPad 고객리스트, 고객카드, 지도, 스케줄, 기록, 설정
- CSV·연락처·사진 OCR 가져오기와 열 매핑
- 고객별 전화·문자·방문·텍스트/사진/음성 메모
- 프로세스 히트맵과 상태·색상 설정 백업
- Google Drive 전체·선택 백업 및 복원
- Shortcuts 기반 단체문자 캠페인과 예약 알림
- TestFlight 배포용 아이콘, 빌드 번호, 수출 규정 설정

미포함 기능:

- 고객리스트 보관함과 복원
- 고객리스트 영구삭제 tombstone
- 관리 기간과 기간 마감
- 날짜·리스트별 내 활동 보고서

## 1.0 (4)

업로드 소스:

```text
브랜치: feature/group-sms
Git 커밋: d44a03674b60ec95f7a50f3cf916b6d5dd721fea
프로젝트 빌드 번호: 4
Git 태그: testflight/1.0-build-4
```

배포 상태:

- Xcode 27 beta 2로 만든 첫 아카이브는 2026-07-18 App Store Connect 검증에서 지원 기간이 지난 beta SDK로 거절됨
- Xcode 26.6과 iOS 26.5 SDK에서 코어 테스트 27개 통과
- Xcode 26.6 컴파일러 호환성을 위해 OCR 점수식의 이름 가림과 복잡한 표현식을 동작 변경 없이 정리
- Xcode 26.6 안정판으로 아카이브하고 2026-07-18 App Store Connect 업로드 성공
- Apple의 빌드 처리가 끝나 `내부 기능 검증`과 외부 `베타 테스트` 그룹에 빌드 4 배정 완료
- 외부 그룹 자동 공지를 활성화하고 빌드 4가 `테스트 중` 상태로 전환됨
- 실제 업로드 소스를 `testflight/1.0-build-4` 태그로 고정하고 GitHub에 푸시

추가 기능:

- 고객리스트를 데이터 손실 없이 보관하고 보관함에서 복원
- 리스트 이름 변경과 고객·연결 기록 전체 이동
- 영구삭제 전 영향 건수 표시 및 앱 전용 파일 정리
- Drive 병합에서 삭제한 리스트가 되살아나지 않도록 tombstone 저장
- 분기·반기·캠페인 단위 관리 기간 생성, 수정, 마감, 보관
- 임의 날짜 범위와 고객리스트별 활동 조회
- 전화·문자·방문·메모·상태 변경·스케줄·고객관리 활동 요약

확인할 테스트:

- 보관한 리스트가 기본 고객 화면에서 숨겨지는지
- 복원 후 고객과 모든 기록이 유지되는지
- 영구삭제 영향 건수와 확인 문구가 정확한지
- 관리 기간 종료일의 활동이 포함되는지
- 기존 전화·문자·방문 기록이 활동 보고서에 한 번만 집계되는지
- Drive 전체 백업 후 다른 기기에서 관리 기간과 삭제 상태가 유지되는지

## 안전한 원복

### 소스만 비교하거나 과거 버전을 빌드할 때

기존 브랜치를 되돌리지 않고 태그에서 별도 브랜치를 만듭니다.

```bash
git switch -c rollback/1.0-build-3 testflight/1.0-build-3
```

### 최신 브랜치를 이전 기능 수준으로 되돌릴 때

공유 브랜치에 `git reset --hard`를 사용하지 않습니다. 되돌릴 변경을 확인한 뒤 `git revert`로 역방향 커밋을 남기거나, 이전 태그에서 수정 브랜치를 만든 뒤 검증하여 병합합니다.

### TestFlight에 이전 코드를 다시 배포할 때

- 기존 빌드 3이 만료되지 않았고 테스트 그룹에 남아 있으면 해당 빌드를 사용할 수 있습니다.
- 다시 업로드해야 하면 빌드 3 태그의 코드를 사용하되 빌드 번호는 새 번호로 올립니다. 이미 성공한 `1.0 (3)` 번호는 재사용하지 않습니다.
- 빌드 4에서 생성한 데이터를 정확한 빌드 3 코드로 열고 다시 저장하면 관리 기간·활동 이벤트·tombstone 같은 신규 필드가 유실될 수 있습니다.
- 데이터가 존재하는 기기에서 원복하기 전에는 빌드 4의 전체 백업을 먼저 만들고, 가능하면 신규 필드 저장 코드는 유지한 호환 수정 빌드를 배포합니다.

## 배포 기록 규칙

모든 TestFlight 업로드는 다음 순서로 기록합니다.

1. 제품 소스와 문서를 커밋하고 원격 브랜치에 푸시합니다.
2. 앱 버전과 빌드 번호를 결정합니다.
3. 릴리스 대장에 기능, 제한, 데이터 호환성, 테스트 항목을 작성합니다.
4. Release 아카이브와 업로드를 완료합니다.
5. App Store Connect 처리 완료와 테스트 그룹 배정을 확인합니다.
6. 실제 업로드 소스에 `testflight/<버전>-build-<번호>` 태그를 생성하고 푸시합니다.
7. 이 문서의 상태, 날짜, 최종 커밋을 갱신합니다.
